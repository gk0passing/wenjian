1. 工作基本信息

UniRec-0.1B 的论文题目是 “UniRec-0.1B: Unified Text and Formula Recognition with 0.1B Parameters”，arXiv 版本提交于 2025 年 12 月，作者包括 Yongkun Du、Zhineng Chen、Yazhen Xie、Weikang Bai、Hao Feng、Wei Shi、Yuchen Su、Can Huang、Yu-Gang Jiang，作者单位主要来自 复旦大学和字节跳动。这里的 0.1B 参数指的是约 1 亿参数，不是一千万参数。

它的任务定位是：

用一个轻量级模型统一识别文档中的文本、公式以及文本-公式混合内容，并支持字符、词、行、段落、多段落 / 文档等多层级输入。

OpenOCR 官方文档也把 UniRec-0.1B 定义为一个中英文统一识别模型，支持普通文本、单行 / 多行数学公式，以及文本公式混合内容。

2. 背景与动机

文档解析里，文本和公式是最核心的信息元素。传统 OCR / 文档解析系统通常把文本识别、公式识别、版面分析、表格识别拆成多个模块；而近两年 VLM 文档解析模型，比如 DeepSeek-OCR、dots.ocr、PaddleOCR-VL、MinerU2.5、Dolphin、MonkeyOCR 等，开始尝试用统一 VLM 做端到端解析。UniRec-0.1B 论文认为，这类统一 VLM 虽然概念上优雅，但通常参数量大、推理成本高，在真实业务里会带来明显延迟。

论文给了一个很关键的观察：在 OmniDocBench 中，文本和公式区域数量占页面区域的 97.43%；使用原始 MinerU2.5 时，文本和公式识别消耗了 87.90% 的总解析时间。也就是说，文档解析的主要耗时不一定在 layout，而是在大量文本块和公式块识别上。UniRec-0.1B 的动机就是：不一定要用大 VLM 识别所有内容，可以用一个 0.1B 的轻量统一识别模型专门加速文本和公式识别。

3. 论文要解决的核心问题

UniRec-0.1B 不是单纯做 OCR 文本行识别，也不是单纯做公式识别，而是要同时解决三个维度的统一：

1. 模态统一：
   文本 + 公式 + 文本公式混合内容

2. 层级统一：
   字符、词、行、段落、多段落 / 文档

3. 语言统一：
   中文 + 英文

但是小模型做这种统一识别会遇到两个关键难点。

第一个是 structural variability，结构层级差异。字符、词、行、段落、多段落的空间结构差异很大。如果模型只把所有标签看成一个 flat sequence，就很难理解“哪里是行内换行，哪里是段落结束，哪里是多段落结构”。

第二个是 semantic entanglement，文本和公式语义纠缠。例如 sum、left、right、frac、infty 这些 token，在普通英文文本和 LaTeX 公式里含义完全不同。如果文本和公式共用同一套 tokenizer，小模型容量有限，很容易混淆“自然语言 token”和“公式符号 token”。论文认为大 LLM 可能靠大容量缓解这种歧义，但 0.1B 小模型会非常敏感。

4. 核心贡献

UniRec-0.1B 的贡献可以概括成四点。

第一，构建了 UniRec40M 数据集，包含约 4000 万条中英文文本、公式、文本公式混合样本，覆盖多层级、多语言、多领域，为轻量统一识别模型提供训练基础。

第二，提出 Hierarchical Supervision Training，HST，分层监督训练，通过在标签序列中显式插入行级和段落级结构 token，让模型学习多行、多段落结构。

第三，提出 Semantic-Decoupled Tokenizer，SDT，语义解耦 tokenizer，分别训练文本 tokenizer 和公式 tokenizer，再把公式 token 作为特殊 token 合入文本 tokenizer，从而让文本语义和公式语义拥有不同 embedding，避免小模型混淆。

第四，构建 UniRec-Bench，从 OmniDocBench 扩展出一个更细粒度的 block-level benchmark，覆盖文本、公式、混合内容、多层级、中英文和九类文档领域，用于评估不同模型在细粒度识别上的能力。

5. UniRec40M 数据集

UniRec40M 是这篇工作的基础。它不是简单收集 OCR 图片，而是围绕“文本 + 公式 + 混合内容 + 多层级结构”专门构建。

数据来源主要有三类。

第一类是 online TeX source。作者收集 arXiv TeX 源文件和 Wikipedia HTML 页面，并转成 TeX 格式。然后给每个有效文本或公式 token 插入不同颜色的 LaTeX color command，再渲染成 PDF。通过颜色和 TeX 源之间的对齐，可以自动得到 word-level、line-level 标签，paragraph-level 标签则通过解析 LaTeX 语法获得。这个流程使得大规模、多层级、文本公式混合标注成为可能。

第二类是 digital-born PDF documents，包括行业研究报告、报纸等，通过 PyMuPDF 抽取文本块及其图像区域，主要补充多层级、多领域的文本识别数据。

第三类是 公开数据集和补充数据，包括 LSVT、MTWI、HierText、CASIA-HWDB、TAL、K-12 等。此外，论文还使用 Qwen3VL-235B-A30B 标注并人工精修了一部分手写笔记数据，作为高质量手写补充。

规模上，UniRec40M 约包含 3000 万英文样本和 1000 万中文样本；按内容类型划分，包括约 1900 万纯文本样本、1300 万纯公式样本和 800 万文本公式混合样本。

6. 模型结构

UniRec-0.1B 采用标准 encoder-decoder 框架。

整体流程可以写成：

输入图像块 / 文档区域
   ↓
保持原始长宽比的 native resolution resize
   ↓
FocalNet / FocalSVTR 类视觉编码器
   ↓
二维视觉特征 flatten 成 visual tokens
   ↓
Transformer decoder cross-attention 到视觉 token
   ↓
自回归生成文本 / 公式 / 混合序列

论文中说明，输入图像采用 native resolution strategy，即保持原始长宽比，同时最大宽度和高度分别限制在 960 和 1408 像素。视觉编码器使用 FocalNet 生成 dense visual feature map，然后把空间维度 flatten 成视觉 token 序列，送入后续多模态 decoder。

Decoder 由 6 层 Transformer 构成，包含 cross-attention，通过 causal mask 自回归预测下一个 token。训练目标是标准 cross-entropy loss。最大 token 长度为 1024，词表大小为 56371。模型从零训练，不加载预训练模型；训练使用 AdamW、batch size 64、10 epochs，并使用旋转、畸变、motion blur、Gaussian noise 等数据增强，训练大约需要 8 张 A800 40GB GPU 上 80 小时。

7. 方法一：HST 分层监督训练

HST 的核心思想是：让模型显式知道文本结构层级，而不是把所有内容当成普通字符序列。

论文在标签序列中加入两个特殊 token：

<|ln|>：line break，表示段落内部换行
<|pn|>：paragraph end，表示段落结束

训练时，这些 token 和普通文本 / 公式 token 一起作为生成目标。这样 decoder 不仅学习“识别什么字符”，还学习“这些字符属于哪一行、哪一段”。推理时，预测出的 <|ln|> 会被移除，<|pn|> 会被替换成两个换行符，用来恢复段落结构。

这个设计特别适合段落、多段落、试卷、论文、报告这种多行文档块。它不是额外加一个 layout head，而是直接把结构信息编码到序列监督里，工程上比较简单。

8. 方法二：SDT 语义解耦 tokenizer

SDT 是这篇工作里非常关键的设计。

普通 tokenizer 往往在文本和公式混合语料上一起训练。问题是，公式里的 \sum、\frac、\left、\right、\infty 和普通英文里的 sum、left、right 不是同一种语义。如果它们共享 token embedding，小模型就要自己学会区分“这是自然语言”还是“这是公式命令”。这对 0.1B 模型来说负担很重。

SDT 的做法是：

1. 单独训练文本 tokenizer
2. 单独训练公式 tokenizer
3. 将公式 tokenizer 中的 token 作为 special tokens 合入文本 tokenizer
4. 已经存在于文本 tokenizer 中的 token 不直接复用，而是保留模态差异

这样 sum 作为英文单词和 \sum 作为数学符号会拥有不同 token / embedding，公式 token 和文本 token 的语义空间被显式解耦。论文认为，这可以从源头减少小模型在文本和公式混合场景下的语义歧义。

9. 实验结果概要
9.1 UniRec-Bench 结果

UniRec-Bench 使用 edit distance 作为主要指标，数值越低越好。它覆盖文本、公式、混合内容、字符 / 词 / 行 / 段落 / 多段落、中英文以及九类文档领域。

在 UniRec-Bench 上，UniRec-0.1B 的整体平均 edit distance 为 0.100，与 PaddleOCR-VL-0.9B 的 0.100 持平，但参数量只有它的约 11%。在细粒度层级上，UniRec-0.1B 的 character / word / line edit distance 分别为 0.025 / 0.032 / 0.043，优于表中多数文档解析模型。

和文本识别专家模型相比，UniRec-0.1B 在字符、词、行等多层级中英文识别上超过 PP-Recv5 和 OpenOCR-Rec；和公式专家模型相比，论文报告其相对 Mathpix、Pix2Tex、UniMERNet-B 分别提升 18.8% / 20.3% / 10.4%。

和 Dolphin-1.5 相比，UniRec-0.1B 在文本、公式、混合内容上分别提升 1.2% / 23.1% / 7.4%；论文认为公式识别的大幅提升主要来自 SDT 对文本和公式 token 的语义解耦。

9.2 领域结果

UniRec-Bench 还按领域评估，包括 Book、PPT2PDF、Research Report、Textbook、Exam Paper、Magazine、Literature、Note、Newspaper 等。UniRec-0.1B 在 Research Report、Exam Paper、Literature、Note、Newspaper 等领域表现很强；论文特别指出，在 Note 手写笔记和 Newspaper 报纸这种视觉噪声较大的场景，UniRec-0.1B 相比很多模型更稳。

这点很有价值，因为很多文档解析模型在数字原生 PDF 上表现不错，但遇到手写、扫描、模糊、老报纸、拍照文档时会明显退化。UniRec40M 里加入手写笔记和报纸类数据，说明它的强项部分来自更全面的数据覆盖。

9.3 OmniDocBench 结果

在 OmniDocBench 上，论文把 UniRec-0.1B 集成到 MinerU2.5 和 PaddleOCR-VL 这类两阶段文档解析系统中，保留它们的 layout analysis 和 table recognition，只替换文本 / 公式 recognition 模块。

结果显示，MinerU2.5 的 full-page edit distance 从 0.143 降到 0.120，页面解析时间从 42.72s 降到 6.2s，接近 7 倍加速。在 PaddleOCR-VL 上，替换识别模块后 full-page edit distance 从 0.115 进一步降到 0.113，也获得速度加速。

这说明 UniRec-0.1B 最强的落地方式不是单独作为完整 parser，而是作为高效文本 / 公式 recognition expert，接在 layout detector 后面替换大 VLM 识别模块。

9.4 推理速度

论文在单张 A800 40GB GPU 上比较推理速度，所有模型都不使用推理加速，batch size 为 1。

模型	参数量	Block Avg	Page Avg
MonkeyOCR	3B	3.62s	58.39s
DeepSeek-OCR	3B-A0.5B	3.47s	58.95s
MinerU2.5	1.2B	2.54s	42.72s
PaddleOCR-VL	0.9B	1.88s	31.92s
Dolphin-1.5	0.3B	0.78s	13.16s
UniRec-0.1B	0.1B	0.37s	6.20s

所以 UniRec-0.1B 的速度优势非常明显，论文总结为 2–9 倍推理加速。

10. 消融实验分析

消融实验主要验证 HST 和 SDT。

去掉 HST 后，文本、公式、混合内容 edit distance 分别变差。加入 HST 后，论文报告文本提升 1.2%，公式提升 1.0%，混合内容提升 1.5%；在五个层级上分别有 0.5%、0.8%、0.4%、1.7%、1.2% 的提升，段落和多段落收益更明显。

去掉 SDT 后，公式识别下降尤其明显。论文报告 SDT 带来文本 1.2%、公式 11.1%、混合内容 1.8% 的提升，说明文本 / 公式 token 解耦对公式识别非常关键。HST 和 SDT 结合后，各领域平均提升 2.4%。

从消融结果看，HST 主要解决结构层级问题，尤其对段落、多段落、多行文档块有效；SDT 主要解决文本公式语义混淆问题，尤其对公式和混合内容有效。

11. 深度分析
11.1 UniRec-0.1B 的本质：不是“大而全 VLM”，而是“小而专的统一识别器”

UniRec-0.1B 和通用 VLM 的路线不同。它没有试图让一个大模型同时理解 layout、表格、图表、问答、推理、全文解析，而是抓住文档解析里最耗时、占比最高的部分：文本和公式识别。论文数据显示，文本和公式在 OmniDocBench 中占页面区域数量的 97.43%，并消耗 MinerU2.5 解析时间的 87.90%，因此优化这部分就能显著加速整个文档解析系统。

所以它的核心价值是：

把文档解析中最常见、最耗时的文本/公式识别任务，
从大 VLM 中拆出来，
用一个 0.1B 小模型做得更快、更稳。

这和 PP-OCRv6、SVTRv2 的“小模型专家化”思路是一致的，但 UniRec 更偏文档解析和公式混合识别，而不是普通场景文本识别。

11.2 为什么它不用 CTC，而用 encoder-decoder？

前面 SVTRv2 证明 CTC 在场景文本识别里可以很强，但 UniRec-0.1B 的任务比普通文本行识别复杂很多。它需要输出：

多行文本
段落结构
多段落结构
LaTeX 公式
文本公式混排
换行、段落、公式 token

这类输出天然更像 sequence generation，所以 UniRec-0.1B 使用 encoder-decoder 架构是合理的。相比 CTC，它更容易表达结构 token、公式命令和跨行关系；相比大 VLM，它又把模型规模控制在 0.1B，从而获得速度优势。

可以理解为：

SVTRv2：
适合文本行 / 场景文本识别，追求极致速度和 CTC 并行解码。

UniRec-0.1B：
适合文档块级文本 + 公式识别，需要生成结构化序列，所以采用轻量 encoder-decoder。
11.3 HST 的价值：把 layout 结构变成序列监督

很多 OCR 模型只关心字符对不对，但真实文档解析还关心：

哪里换行
哪里分段
公式是在行内还是独立显示
多行公式是否保持正确换行

HST 不需要额外训练一个 layout head，而是通过 <|ln|> 和 <|pn|> 把结构标签融入序列生成目标。这种设计的好处是简单、统一、训练稳定；缺点是结构表达能力仍然有限，主要适合 line / paragraph 级别，对复杂表格、跨栏阅读顺序、图文混排仍然需要外部 layout 模块配合。

11.4 SDT 是小模型做公式识别的关键

公式识别和文本识别最大的差别在于：公式不是自然语言，它有强语法结构和符号语义。比如：

\frac{a}{b}
\sum_{i=1}^{n}
\left( x + y \right)

这些 token 在普通英文中可能只是普通词或符号，但在 LaTeX 中有明确的结构意义。大 VLM 可以依靠海量参数和上下文能力缓解歧义，小模型如果 tokenizer 混在一起，会把很多公式 token 学成“自然语言 token 的另一种用法”。SDT 通过独立 tokenizer 和特殊 token 机制，把这种歧义提前消掉，因此对公式识别提升特别明显。

这对工业 OCR 很有启发：小模型不一定要靠增大参数解决语义歧义，有时候把标签空间、token 空间、类别空间设计清楚，收益会更大。

12. 局限性

第一，UniRec-0.1B 原始版本主要支持文本和公式识别，不是完整文档解析系统。OpenOCR 文档中提到，OpenDoc-0.1B 后续重建 UniRec-0.1B，才扩展到文本、公式和表格统一识别；而原始 UniRec-0.1B 只支持文本和公式。

第二，它用于完整页面解析时，仍然依赖 layout analysis。论文在 OmniDocBench 上的实验是保留 MinerU2.5 或 PaddleOCR-VL 的 layout analysis 和 table recognition，只替换文本 / 公式识别模块。因此它更像强 recognition expert，而不是完全替代 layout parser 的端到端模型。

第三，它是 autoregressive decoder，虽然比大 VLM 快很多，但对纯文本行 OCR 来说，速度未必能超过 CTC-based recognizer。因此如果业务只做单行文字识别，SVTRv2 / PP-Rec 类 CTC 模型可能仍然更合适；如果业务包含多行、段落、公式、混排，UniRec-0.1B 更有优势。

第四，论文附录提到 UniRec-0.1B 的失败案例之一是：模型经常把数学上下文之外的上下标预测成普通文本，这与基于 PyMuPDF 的训练数据抽取方式有关，因为其中的 subscript / superscript 元素被表示成 plain text。
